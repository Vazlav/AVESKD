Учетная политика конфигурации


Учет товаров на оптовых складах ведется в закупочных ценах.

Учет товаров на аптеках ведется в розничных ценах (и закупочных параллельно).


Товар учитывается по параметрам СУММА и КОЛИЧЕСТВО.


Учет товаров ЖНВЛС ведется по партиям, они обклеиваются штрихкодами. Таким образом по товарам ЖНВЛС всегда известна закупочная стоимость каждой конкретной упаковки.


Товар НЕ жнвлс учитывается по средней стоимости. При движении товара СУММА остатка товара делится на КОЛИЧЕСТВО ОСТАВШЕГОСЯ и умножается на КОЛИЧЕСТВО НЕОБХОДИМОЕ. При этом, если результат деления имеет более 2-х знаков после запятой разница корректируется последней единицей товара.


Политика ценообразования


В качестве перечня номенклатуры выступает справочник Ассортиментный План.В нем задаются параметры товаров и минимальные и максимальные цены по всей сети.


В справочнике АП торговой точки задаются цены для конкретной точки, а также минимальное и максимальное количество товара на точке, для расчета автоматического заказа.



Политика учета товара по ставкам НДС


Для ускорения получения отчетов по товару в разрезе ставок НДС ведется параллельный учет в отдельном регистре. Учет по этому регистру СУММОВОЙ, без количества. Так же этот регистр используется для выгрузки данных в бухгалтерию.


Почему не должны вылезать копейки


ИБО...

  1. Поступление товара идет как Колво Сумма => т.о. Удается получить документ всегда совпадающий с документом поставщика

  2. На розничном складе товар числится по РОЗНИЧНОЙ СУММЕ = колво товара * розничную цену, назначенную на каждую единицу товара. => т.о. Сумма розничная товара ВСЕГДА кратна его количеству. Если товар переоценивается – то тоже весь остаток * новую цену-> тоже сумма кратна количеству.

  3. Фискальный регистратор продает товар по алгоритму Цена товара * количество => розничная сумма всегда списывается в 0.

  4. Закупочная сумма списывается по алгоритму Сумма/КолвоОст*КолвоНадо и если остался хвост корректировка последней единицей => т.о. Тоже списывается в 0.


Переоценка


Работает в 3 этапа:

  1. Создается документ-команда на переоценку

  2. Отправляется на аптеку. Там происходит переоценка товара имеющегося в наличии на момент ФАКТИЧЕСКОЙ переоценки (читай пересчета, переклейки ценников если необходимо, перераспечатки ценников для полок и т.д....), т.к. На момент переоценки в офисе не всегда известно количество оставшегося товара и тормозить продажи ради переоценки не логично.

  3. Вместе с выгрузкой, с аптеки приходит документ ФАКТИЧЕСКАЯ переоценка, в котором содержится количество РЕАЛЬНО переоцененного товара и собственно суммы переоценки.


Взаиморасчеты с поставщиками


Ведутся в одельном регистре. Данные об оплатах загружаются из бухгалтерии. Это позволит избавить менеджеров от необходимости шариться в бухгалтерской базе.


Расколбас данных между базами


Вариант 1. Производить при помощи отдельного SQL сервера, на котором будет крутиться некая база с таблицами обмена. Поскольку аптеки подключаются по VPN можно организовать им доступ к этому серверу и обмен данными организовать в виде прямых запрсов к серверу-обменщику. Параллельно предусмотреть вариант файлового обмена.


Вариант 2. Опять же Аптеки на VPN. Из офиса доступны их SQL сервера. Обмен производить через сервера аптек...
